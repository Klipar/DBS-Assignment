\section{Physical Model - Implementation}

\subsection{List of Tables}
The following tables have been implemented to represent the cinema management system:

\begin{itemize}
    \item \textbf{staffs}: Stores employee information, including names, age, and professional positions.
    \item \textbf{halls}: Contains data regarding cinema auditoriums, their names, and seating capacities.
    \item \textbf{films}: A catalog of movies featuring titles, durations, and age ratings.
    \item \textbf{actors}: A registry of actors (first and last names) performing in the films.
    \item \textbf{genres}: A lookup table for various movie genres available in the system.
    \item \textbf{studios}: A registry of production studios responsible for the films.
    \item \textbf{screenings}: Scheduled movie showtimes, linking films to specific halls and times.
    \item \textbf{shifts}: Employee work schedules, including descriptions and time intervals.
    \item \textbf{sales}: Records of ticket transactions, linking customers to specific screenings.
    \item \textbf{actors\_films}: Junction table managing the M:N relationship between actors and films.
    \item \textbf{films\_genres}: Junction table for assigning multiple genres to films (M:N).
    \item \textbf{films\_studios}: Junction table linking films to their respective production studios.
    \item \textbf{shifts\_screenings}: Junction table assigning staff shifts to the screenings they service.
\end{itemize}

\subsection{Notable Integrity Constraints}
The following constraints were implemented to enforce specific business rules and ensure data validity:

\begin{itemize}
    \item \textbf{Employee Age (CHECK)}: The constraint \texttt{age >= 15} in the \texttt{staffs} table ensures that the cinema complies with labor regulations allowing the employment of teenagers from the age of 15.
    \item \textbf{Auditorium Capacity (CHECK)}: The \texttt{capacity > 0} condition in the \texttt{halls} table prevents the creation of invalid rooms with no seating.
    \item \textbf{Film Duration (CHECK)}: The \texttt{duration} field must be a positive integer, eliminating data entry errors for movie lengths.
    \item \textbf{Genre Uniqueness (UNIQUE)}: Ensures that genre names are not duplicated, maintaining a clean and consistent categorization system.
    \item \textbf{Referential Integrity}: All foreign keys ensure that child records (e.g., sales, screenings) always point to existing parent records, preventing orphaned data.
\end{itemize}

\subsection{Selected Referential Actions}
To maintain database consistency and prevent accidental data loss, the following strategies were applied to all foreign key relationships:

\begin{itemize}
    \item \textbf{ON DELETE RESTRICT}: This strategy is used across all tables to prevent the deletion of a parent record if any referenced child records exist. For example, a film cannot be deleted if there are existing screenings or sales associated with it. This ensures the preservation of historical data and prevents "orphaned" records that would break the system's logic.
    \item \textbf{ON UPDATE CASCADE}: This strategy ensures that any changes to primary keys (e.g., updating a unique identifier) are automatically propagated to all related foreign keys in child tables. This maintains seamless connectivity between entities without manual intervention after an update.
\end{itemize}

\section{Searching for information about films}
\subsection{Process Description}

This process implements User Scenario 2, where a customer seeks detailed information about films based on
specific criteria. The process involves a two step interaction:

\begin{itemize}
    \item Filtering: The system filters films based on the actor's name, genre, and production studio. Returns id, name, duration and age group.
    \item Search for a session: Once a film is identified, the system request its technical details: id, time of session start, language, hall name and amount of free seats.
\end{itemize}

\subsection{SQL Implementation}
The implementation consists of two main queries: one for finding the film and another for fetching screening information.

\subsubsection{Query 1: Search by Filters (Actors, Genres, Studios)}
\begin{lstlisting}
-- Parameters:
--   $1 = actor name
--   $2 = actor surname
--   $3 = actor genre
--   $4 = film studio
SELECT DISTINCT
  f.id,
  f.name,
  f.duration,
  f.age_group

FROM
  films AS f

JOIN actors_films AS af ON af.film_id = f.id
JOIN actors AS a ON af.actor_id = a.id AND a.name = '{{name}}' AND a.surname = '{{surname}}'

JOIN films_genres AS fg ON fg.film_id = f.id
JOIN genres AS g ON g.id = fg.genre_id AND g.name = '{{genre}}'

JOIN films_studios AS fs ON fs.film_id = f.id
JOIN studios AS s ON s.id = fs.studio_id AND s.name = '{{studio}}';
\end{lstlisting}

\subsubsection{Query 1: Search by Filters (Actors, Genres, Studios)}
\begin{lstlisting}
-- Parameters:
--   $1 = film id
SELECT
  s.id,
  s.start_time,
  s.language,
  h.name AS hall_name,
  h.capacity - COUNT (sl.id) AS free_seats

FROM
  screenings AS s

JOIN halls AS h on h.id = s.hall_id
LEFT JOIN sales AS sl ON sl.screening_id = s.id
WHERE s.film_id = $1
GROUP BY s.id, h.name, s.start_time, s.language, h.capacity
HAVING h.capacity > COUNT(sl.id)
ORDER BY start_time ASC
\end{lstlisting}

prioritizing the customer's need for instant information over the overhead of infrequent data updates.

\section{Session schedule planning}
\subsection{Process Description}

This process implements User Scenario 4, where cinema staff plans a new screening in the session schedule.
The main goal is to insert a new screening only when the selected hall is available for the whole duration of the film.
The process consists of two steps:

\begin{itemize}
    \item Conflict checking: Before a new screening is inserted, the system checks whether another screening already occupies the same hall during the requested time interval.
    \item Screening insertion: If no overlap is found, the system inserts the new screening with the selected film, hall, start time, and language.
\end{itemize}

\subsection{SQL Implementation}
The implementation consists of two main queries: one analytical query for detecting schedule conflicts and one operational query for inserting a valid screening.

\subsubsection{Query 1: Check Hall Availability}
\begin{lstlisting}
-- Parameters:
--   $1 = film_id of the new screening
--   $2 = hall_id of the new screening
--   $3 = start_time (timestamp) of the new screening
SELECT COUNT(*) AS overlapping_screenings
FROM screenings s
JOIN films existing_film ON existing_film.id = s.film_id
JOIN films new_film ON new_film.id = $1
WHERE s.hall_id = $2
AND (($3::timestamptz), ($3::timestamptz + make_interval(mins => new_film.duration)))
OVERLAPS
(s.start_time, s.start_time + make_interval(mins => existing_film.duration));
\end{lstlisting}

This query compares the requested time interval of the new screening with all existing screenings in the same hall.
The end time is calculated dynamically from the film duration, which prevents situations where two screenings are planned in the same hall at overlapping times.
If the returned value is \texttt{0}, the hall is available and the insert can be performed.

\subsubsection{Query 2: Insert New Screening}
\begin{lstlisting}
-- Parameters:
--   $1 = film_id of the new screening
--   $2 = hall_id of the new screening
--   $3 = start_time (timestamp) of the new screening
--   $4 = language of the new screening
INSERT INTO screenings (film_id, hall_id, start_time, language)
VALUES ($1, $2, $3, $4);
\end{lstlisting}

The insertion query stores the newly planned screening in the \texttt{screenings} table.
It should be executed only after the conflict checking query confirms that there are no overlapping screenings in the selected hall.

\subsection{Supporting Indexes}
To optimize conflict checking during schedule planning, the following index was implemented:
\begin{lstlisting}
CREATE INDEX idx_screenings_hall_start_film
ON screenings (hall_id, start_time, film_id);
\end{lstlisting}

Total execution time was reduced for about 15-30\%.

The performance gain is relatively small because the seeded dataset contains only 1,000 screenings,
which is still a modest amount of data for this type of query.
However, the index is justified because schedule planning is a correctness-critical operation:
each new screening must quickly verify hall availability before insertion.
As the number of planned screenings grows, filtering by hall and start time becomes increasingly important.

For the second query, the INSERT, the index does not provide a direct performance benefit.
Inserting a new row into \texttt{screenings} also requires PostgreSQL to update the related index structure,
so the indexed version can be slightly slower than the version without an index.
In our measurements, the execution times were very small and repeated runs often converged to similar values due to caching.
Therefore, the index is justified primarily by the conflict-checking query, while the possible insert overhead remains acceptable because screenings are inserted much less frequently than they are checked, searched, or displayed in customer-facing workflows.

\subsection{Impact on Operational Insert}
The index used for session schedule planning also affects the operational insert query.
Unlike analytical lookup queries, an \texttt{INSERT} does not benefit directly from an index on the target table.
Instead, PostgreSQL must maintain the index whenever a new row is inserted into \texttt{screenings}.

\section{Evaluation and Conclusion}
\label{sec:evaluation}

The implemented physical database supports the main cinema processes required for this assignment, including film search, session selection, and screening schedule planning.
The selected use cases demonstrate both read-oriented queries and write-oriented operations, together with the indexes needed to support them efficiently.

The most surprising part was the seed data generation.
It was a new experience for us to create a larger artificial dataset while preserving foreign keys and realistic relationships between tables.

During the implementation of the described processes, no bugs were encountered in the original model.
Therefore, the model did not require structural changes for these use cases.

The hardest business rule to express in SQL was checking for overlapping screenings.
This rule was more complex than simple filtering or insertion because the system had to compare two time intervals:
the planned screening interval and the intervals of existing screenings in the same hall.
The use of the \texttt{OVERLAPS} operator made the final query compact, but the logic still required careful thinking about film duration and hall availability.

We used AI to consult on performance optimization and indexing strategies, which helped us understand the trade-offs between read and write performance.
